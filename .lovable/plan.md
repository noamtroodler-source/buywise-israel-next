
Root cause found: the import is failing in the backend, not in the CSV cleaning step.

What is actually happening
- The upload reaches the backend successfully.
- The backend returns `success: true`, but each sub-batch reports an error.
- The live network response shows the exact error:
  `cannot insert a non-DEFAULT value into column "price_per_sqm"`
- I verified the database schema: `sold_transactions.price_per_sqm` is a generated column (`GENERATED ALWAYS AS ...`), so Postgres forbids inserts/upserts that explicitly send a value for it.

Why all 15,252 rows fail
- In `supabase/functions/import-govmap-data/index.ts`, the function builds each record with:
  - `sold_price`
  - `size_sqm`
  - `price_per_sqm: Math.round(txn.sold_price / txn.size_sqm)`
- Because `price_per_sqm` is generated by the database, every single write in every batch is rejected.
- That perfectly matches the UI: 0 imported, 0 skipped, all failed.

Secondary findings
- The earlier auth issue is not the current blocker; the function is receiving authenticated requests.
- The unique constraint for `(address, city, sold_date, sold_price)` does exist, so conflict handling is not the current failure.
- There is also a harmless UI warning in `ImportGovMapData`: the custom `Stat` component is likely being used inside a component that passes refs, and it does not use `forwardRef`. This is unrelated to the failed import.

Implementation plan
1. Fix the import backend
- Remove `price_per_sqm` from the payload written to `sold_transactions`.
- Let the database compute it automatically from `sold_price / size_sqm`.
- Keep the rest of the import logic unchanged.

2. Harden the backend error reporting
- Return clearer batch-level diagnostics from `import-govmap-data`, including batch number and first failing record metadata when a batch fails.
- This makes the next failure, if any, immediately visible without guesswork.

3. Improve the admin upload UI feedback
- In `src/pages/admin/ImportGovMapData.tsx`, surface the backend `errors[]` array in the page and/or toast instead of only incrementing failed counts.
- Show a short “first backend error” message under upload progress so failures are understandable in one glance.

4. Fix the noisy React warning
- Refactor the `Stat` component usage so it does not receive a ref improperly, or convert it to `forwardRef` only if needed by the surrounding UI primitive.
- This is not blocking import, but cleaning it up will remove distracting false leads during debugging.

5. Re-test the full flow
- Re-run Parse & Clean.
- Re-run Upload.
- Confirm the counts now show real imports/skips instead of universal failures.
- Only after import succeeds, test geocoding separately, since it may have its own issues later.
